# Project HAL - Firmware Context

## Overview
The firmware runs on a Wemos D1 Mini (ESP8266) and is responsible for:
- Connecting to WiFi
- Polling the HAL server and acting on responses
- Controlling the relay (load switching)
- Reading physical switch input
- Managing device mode via RGB LED
- Handling pairing mode via WiFi access point
- Reporting diagnostics to the server

## Development Environment

- **Framework:** Arduino (C++)

## Hardware
- **MCU:** Wemos D1 Mini (ESP8266)
- **Power:** HiLink HLK-PM01 (120V AC → 5V DC)
- **Relay:** Isolated relay module, active HIGH, wired to D1 (GPIO5)
- **Switch:** Momentary SPDT center off, low voltage
  - UP terminal → D2 (GPIO4) — always on
  - DOWN terminal → D3 (GPIO0) — always off
  - Common → GND
- **RGB LED:** Common cathode (verify on hardware)
  - R → D5 (GPIO14) via 220Ω resistor
  - G → D6 (GPIO12) via 220Ω resistor
  - B → D7 (GPIO13) via 220Ω resistor
  - Common → GND

## GPIO Summary
D1 (GPIO5)  — relay control (output, active HIGH)
D2 (GPIO4)  — switch UP / always on (input)
D3 (GPIO0)  — switch DOWN / always off (input)
D5 (GPIO14) — RGB red (output, PWM)
D6 (GPIO12) — RGB green (output, PWM)
D7 (GPIO13) — RGB blue (output, PWM)
D8 (GPIO15) — pairing button (input, pull down required)

⚠️ GPIO15, and GPIO0 affect boot mode — be aware of their state at boot.
Pairing button on GPIO15 must have pull down resistor.

## Communication Protocol
Device polls server via HTTP GET:
GET /poll?state=1&boot=42&reason=0&uptime=3600
X-API-Key: unique-per-device-key

Response is a single byte:
- `0` — no change
- `1` — toggle relay

Query parameter reference:
- `state` — current relay state (0=off, 1=on)
- `boot` — boot count (persisted in flash)
- `reason` — reset reason code from ESP.getResetInfoPtr()
- `uptime` — seconds since last boot via millis()/1000

## Device Modes
CONNECTED          — solid green LED, polling server, relay operational
CONNECTING         — green blinking LED, attempting WiFi connection
WIFI_OFFLINE       — red/blue blinking LED, WiFi connection lost
SERVER_UNREACHABLE — red/green blinking LED, server not responding
PAIRING            — solid blue LED, AP mode, captive portal active (Phase 4)

## Pairing Mode
- Triggered automatically if no credentials in flash
- Triggered manually by holding pairing button for 3 seconds
- Uses WiFiManager library to handle AP and captive portal
- Captive portal collects:
  - WiFi SSID (pre-filled from connected network via iOS app)
  - WiFi password
  - Server URL
  - API key (generated by server, entered via iOS app)
- On successful configuration device reboots into normal mode
- AP times out after 90 seconds if no connection — returns to pairing mode

## Diagnostics
- Reset reason read from ESP.getResetInfoPtr() on boot
- Boot count stored in EEPROM, incremented on every boot
- Uptime calculated from millis()
- All reported on every poll in query parameters
- Unexpected reset reasons (watchdog, exception) flagged by server

## Memory Considerations
- ESP8266 has 80KB usable RAM
- TLS is memory intensive (~20-40KB) — HTTP used for now, upgrade
  to HTTPS if memory allows or if hardware upgraded to ESP32
- ArduinoJson avoided in favor of plain text/query parameters
- Arduino String class used sparingly to avoid heap fragmentation
- If random reboots occur after extended runtime, suspect memory
  pressure — consider ESP32 upgrade

## Debug Mode
- Compiled with -D DEBUG flag in d1_mini_debug environment
- Serial output active at 115200 baud
- All serial output wrapped in DEBUG guards:
```cpp
#ifdef DEBUG
  Serial.println("debug info");
#endif
```
- Remove debug environment before deployment

## C++ Notes for This Project
- Always include `#include <Arduino.h>` at top of every .cpp file
- Use `enum class` over plain `enum` for type safety
- Prefer fixed size buffers and sprintf over Arduino String class
- Use `const char*` over String where possible
- Use namespaces to organize related constants and functions
- Internal pull-ups available on most GPIO pins via
  `pinMode(pin, INPUT_PULLUP)`
