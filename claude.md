# Project HAL - Claude Code Context

## Project Overview
Project HAL is a DIY WiFi enabled smart light switch system spanning 
embedded firmware, a Go backend, and a native iOS app. The name is a 
reference to HAL 9000 from 2001: A Space Odyssey.

> "I'm sorry Dave, I'm afraid I CAN turn that light off"

## System Architecture

### Components
- **Wemos D1 Mini** — ESP8266 based MCU, handles WiFi, relay control,
  and physical switch input
- **HiLink HLK-PM01** — AC-DC power supply, 120V in, 5V out, powers D1 Mini
- **Isolated relay module** — switches 120V load, controlled by MCU GPIO
- **RGB LED** — indicates device mode (green=normal, blue=pairing, red=debug)
- **Momentary SPDT center off switch** — low voltage physical control input,
  wired directly to GPIO pins on D1 Mini

### Data Flow
Physical switch (momentary SPDT, low voltage) → GPIO input (D1 Mini)
↓
MCU logic
↓
Web UI / iOS App → Go server (desired state) ←→ MCU polls server
↓
GPIO output → relay → load

### Communication Protocol
- Device polls server via HTTP GET every 1-2 seconds
- State and diagnostics sent as query parameters (not JSON, to minimize
  memory usage on ESP8266)
- API key sent in X-API-Key header — never in query parameters
- Server responds with single byte: 0 (no change) or 1 (toggle)
GET /poll?state=1&boot=42&reason=0
X-API-Key: unique-per-device-key
Response: 0 or 1

### Authentication & Security
- **Users** authenticate with JWT
- **Devices** authenticate with unique per-device API keys
- Device has no concept of user accounts — it only knows its API key
- Server associates devices to users internally
- API keys generated by server during pairing, stored in device flash
- Internet facing server requires HTTPS

### Device State Model
- **Current state** — reported by device on every poll
- **Desired state** — set by web UI or iOS app
- Server compares the two and tells device to toggle if they differ
- Device is single source of truth for current state
- Server is single source of truth for desired state

### Pairing Flow
1. User creates new device in iOS app — server generates unique API key
2. User holds pairing button on device for 3 seconds
3. Device enters pairing mode — RGB LED turns blue, AP spins up
4. iOS app uses NEHotspotConfiguration to join HAL access point
5. App POSTs WiFi credentials and API key to 192.168.4.1/configure
6. Device saves credentials to flash, reboots into normal mode
7. iOS app rejoins home network, device appears active in app

### Diagnostics
- Device reports reset reason, boot count, and uptime on every poll
- Server logs unexpected reboots (watchdog, exception, soft watchdog)
- Boot count persisted in flash across reboots
- Visible in web UI as device health log
